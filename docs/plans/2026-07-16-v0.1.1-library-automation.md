# Music Manager v0.1.1 Library Automation Implementation Plan

> **For Hermes:** Execute through the named-team Kanban pipeline. L establishes/reviews the specification; Light implements with TDD; Jarvis integrates, verifies, and controls release.

**Goal:** Turn v0.1.0 into a safe authenticated manager that can stage, identify, tag, deduplicate, atomically import, monitor, and upgrade releases while hardening providers and packaging.

**Architecture:** Add explicit acquisition/release/import state models and isolate filesystem mutation behind a transactional import service. Downloads enter staging, matching resolves recording plus edition identity with scored evidence, and imports use destination-filesystem temporary files plus atomic rename, verification, collision protection, and rollback.

**Tech Stack:** Python 3.12, FastAPI, async SQLAlchemy/SQLite, Alembic, Jinja2, pytest, Ruff, mypy, Mutagen, Chromaprint/AcoustID, Docker.

## Acceptance criteria and invariants

1. Version target is **v0.1.1**.
2. Naming remains exactly `{album_artist}/{year} - {album}/{disc_track} - {title}.{ext}`; one disc uses `TT`, multiple discs use `D-TT`, with two-digit tracks.
3. Unicode and existing traversal/reserved-name/path-length/extension defenses remain intact.
4. Imports never silently overwrite. Failures leave the current library unchanged and staged input retryable/reviewable.
5. Cross-filesystem movement is not called atomic. Use same-filesystem rename, or copy to a hidden destination-filesystem temp file, flush/fsync, validate, atomically rename, then remove source only after success.
6. Fingerprints identify recordings, not editions. Edition selection also uses release MBID, medium/track position, duration, track count, year/country/label/catalog/barcode where available, plus confidence thresholds.
7. Duplicate/collision decisions use content hash, fingerprint/recording identity, release/track identity, and destination path. Ambiguity goes to review.
8. Tag writes are format-aware, idempotent, and verified by readback before import completion.
9. Monitoring/upgrades cannot replace a preferred file until candidate matching, tagging, collision, and quality checks pass.
10. YouTube remains explicitly volatile; require timeouts, cancellation, cookie/auth health, provenance, and actionable failures.
11. TIDAL may only use a real lawful authenticated external downloader/API available to the operator. No DRM circumvention, credential scraping, simulation, or false availability. Without a viable backend, keep live acquisition blocked and state exact prerequisites.
12. Multi-user auth uses strong password hashes, secure sessions, CSRF protection, and owner/admin versus standard-user authorization.
13. Eliminate the pytest-asyncio deprecation warning with supported explicit configuration and warning-clean tests.
14. Wheel and sdist include Jinja/static assets and pass a clean-installed-artifact dashboard/static smoke test.
15. Do not restart/recreate Unraid containers or publish release/latest Docker tags without Caleb/Jarvis approval.

## Required state separation

- Acquisition: `queued`, `searching`, `acquiring`, `downloaded`, `failed`, `cancelled`.
- Import workflow: `discovered`, `staged`, `matching`, `needs_review`, `ready`, `importing`, `imported`, `failed`, `rolled_back`.
- Persist candidate evidence, source status, staging paths, hashes/fingerprints, recording/release IDs, edition attributes, quality, match score/reasons, planned operations, collision/tag verification, error/rollback detail, and monitoring history.

## Slice 2 — Edition matching (Light after L approval)

Create matching/evidence services with deterministic scores and human-readable reasons. Unattended matching requires recording and release evidence agreement. Contradictory/incomplete/tied candidates enter `needs_review`; AcoustID alone cannot select an edition; manual selection is auditable.

**Tests:** original/deluxe/reissue, multi-disc, duration drift, compilation artist, missing release MBID, contradictory barcode/catalog evidence, ties.

**Commit:** `feat(matching): add evidence-based edition resolution`.

## Slice 3 — Duplicate/collision/tag/atomic import (Light after L approval)

Create import planner/executor/collision and tag writer services; add Mutagen; add API/UI review surfaces. Plan before mutation, avoid link escape, hash/fingerprint, tag and read back, use destination temp plus atomic rename/fsync, recheck destination before commit, roll back the release on failure, retain staged sources, and verify count/hash/path/tag before `imported`.

**Tests:** same path/different bytes, same bytes/different source, cross-edition duplicate, destination race, symlink destination, disk-full/short-copy, tag readback failure, cross-device staging, interruption recovery, rollback after N of M tracks, Unicode and extension preservation.

**Commit:** `feat(import): add verified atomic library import`.

## Slice 4 — Monitoring and quality upgrades

Persist monitoring and quality profiles; schedule non-overlapping cancellable checks; rank codec/lossless/sample-rate/bit-depth/bitrate/channels/reliability/preferences; require meaningful gain and match confidence; reuse the import transaction; preserve prior file until verified replacement; retain history/review/rollback.

**Commit:** `feat(monitoring): add release monitoring and quality upgrades`.
